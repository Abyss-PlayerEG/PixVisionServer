## 概述

PixVision Server 采用了**双层拦截器架构**实现灵活且高效的权限管理系统：

1. **认证层（Authentication）**：JWT Token 验证，确保用户身份合法
2. **授权层（Authorization）**：基于角色的访问控制（RBAC），确保用户有足够权限

### 核心特性

- ✅ **注解驱动**：使用 `@RequireRole` 注解声明式配置权限
- ✅ **细粒度控制**：支持类级别和方法级别的权限控制
- ✅ **继承机制**：方法级注解优先于类级注解
- ✅ **灵活配置**：支持多角色组合和权限继承
- ✅ **高性能**：只对标注注解的接口进行权限检查
- ✅ **完善日志**：记录所有权限验证结果，便于审计和排查

## 架构设计

### 请求处理流程

```
客户端请求
    ↓
┌─────────────────────────────┐
│  JWT 认证拦截器              │ ← 第1层：验证 Token 有效性
│  - 检查 Token 是否存在       │
│  - 验证 Token 是否在白名单   │
│  - 解析用户信息              │
└─────────────────────────────┘
    ↓ (认证成功)
┌─────────────────────────────┐
│  权限验证拦截器              │ ← 第2层：检查角色权限
│  - 检测 @RequireRole 注解    │
│  - 查询用户角色              │
│  - 验证权限是否匹配          │
└─────────────────────────────┘
    ↓ (权限通过)
Controller 方法执行
    ↓
返回响应
```

### 技术栈

- **JWT Token**：用户身份认证
- **Redis 白名单**：Token 有效性管理
- **自定义注解**：`@RequireRole` 声明权限需求
- **Spring Interceptor**：拦截器实现权限验证
- **MyBatis-Plus**：查询用户角色信息

## 快速开始

### 1. 基础用法

在 Controller 方法上添加 `@RequireRole` 注解：

```java
@RestController
@RequestMapping("/api/admin")
public class AdminController {
    
    @GetMapping("/users")
    @RequireRole(value = {77})  // 仅系统管理员可访问
    public ResponsePojo<List<User>> getUsers() {
        return userService.getAllUsers();
    }
}
```

### 2. 类级别权限

为整个控制器设置默认权限：

```java
@RestController
@RequestMapping("/api/admin")
@RequireRole(value = {77})  // 所有接口都需要系统管理员权限
public class AdminController {
    
    @GetMapping("/users")
    public ResponsePojo<List<User>> getUsers() {
        // 继承类的权限：仅系统管理员
    }
    
    @GetMapping("/config")
    @RequireRole(value = {66, 77})  // 覆盖为：工单管理员或系统管理员
    public ResponsePojo<Config> getConfig() {
        // 使用方法级注解覆盖类级权限
    }
}
```

### 3. 无需权限控制

不添加 `@RequireRole` 注解的接口，所有已登录用户均可访问：

```java
@RestController
@RequestMapping("/api/user/profile")
public class UserProfileController {
    
    @PostMapping("/change/nickname")
    // 无注解，所有已登录用户可修改自己的昵称
    public ResponsePojo<Boolean> updateNickname(...) {
        // ...
    }
}
```

## 核心组件

### 1. RequireRole 注解

**位置**：`top.playereg.pix_vision.util.Annotation.RequireRole`

**参数说明**：

| 参数         | 类型      | 必填 | 默认值 | 说明                           |
|------------|---------|:--:|-----|------------------------------|
| `value`    | `int[]` | ✅  | -   | 允许的角色代码数组                   |
| `allowHigher` | `boolean` | ❌  | `true` | 是否允许更高权限的角色访问（权限继承） |

**使用示例**：

```java
// 示例 1：仅允许指定角色
@RequireRole(value = {55}, allowHigher = false)
// 只有审核员（55）可以访问，管理员（77）也不能访问

// 示例 2：允许多个角色 + 权限继承
@RequireRole(value = {22, 77}, allowHigher = true)
// 创作者（22）或系统管理员（77）可以访问
// 如果用户是工单管理员（66），因为 66 > 22，也可以访问

// 示例 3：仅允许指定角色，不允许继承
@RequireRole(value = {66, 77}, allowHigher = false)
// 只有工单管理员（66）或系统管理员（77）可以访问
```

### 2. PermissionInterceptor 拦截器

**位置**：`top.playereg.pix_vision.handler.PermissionInterceptor`

**职责**：
- 检测 `@RequireRole` 注解
- 从 Request 中获取用户 ID
- 查询数据库获取用户角色
- 验证用户角色是否符合权限要求
- 返回 403 错误（权限不足）或放行

**执行时机**：
- 在 JWT 认证拦截器之后执行
- 只处理方法级别的请求
- 只对标注了 `@RequireRole` 的方法生效

### 3. WebConfig 配置

**位置**：`top.playereg.pix_vision.config.WebConfig`

**拦截器注册顺序**：

```java
@Override
public void addInterceptors(InterceptorRegistry registry) {
    // 第1层：JWT 认证拦截器
    registry.addInterceptor(jwtAuthenticationInterceptor)
        .addPathPatterns("/api/**")
        .excludePathPatterns(
            "/api/user/auth/login",      // 登录
            "/api/user/auth/register",   // 注册
            "/api/mail/send-*-code"      // 发送验证码
        );
    
    // 第2层：权限验证拦截器
    registry.addInterceptor(permissionInterceptor)
        .addPathPatterns("/api/**");
}
```

## 使用方式

### 方式一：方法级别权限控制（推荐）

适用于单个接口需要特殊权限控制的场景。

```java
@RestController
@RequestMapping("/api/works")
public class WorksController {
    
    // 所有已登录用户可以浏览作品
    @GetMapping("/list")
    public ResponsePojo<List<Work>> listWorks() {
        return workService.listWorks();
    }
    
    // 仅创作者和管理员可以上传作品
    @PostMapping("/upload")
    @RequireRole(value = {22, 77})
    public ResponsePojo<Boolean> uploadWork(@RequestBody Work work) {
        return workService.uploadWork(work);
    }
    
    // 仅审核员和管理员可以审核作品
    @PostMapping("/approve")
    @RequireRole(value = {55, 77}, allowHigher = false)
    public ResponsePojo<Boolean> approveWork(@RequestParam Integer workId) {
        return workService.approveWork(workId);
    }
}
```

### 方式二：类级别权限控制

适用于整个控制器需要统一权限的场景。

```java
@RestController
@RequestMapping("/api/admin/system")
@RequireRole(value = {77})  // 所有接口仅系统管理员可访问
public class SystemAdminController {
    
    @GetMapping("/config")
    public ResponsePojo<Config> getSystemConfig() {
        // 继承类权限：仅系统管理员
    }
    
    @PostMapping("/restart")
    public ResponsePojo<Boolean> restartService() {
        // 继承类权限：仅系统管理员
    }
}
```

### 方式三：混合使用

类级别设置默认权限，方法级别覆盖特定接口的权限。

```java
@RestController
@RequestMapping("/api/admin")
@RequireRole(value = {66, 77})  // 默认：工单管理员或系统管理员
public class AdminController {
    
    @GetMapping("/tickets")
    public ResponsePojo<List<Ticket>> getTickets() {
        // 继承类权限：工单管理员或系统管理员
    }
    
    @GetMapping("/users")
    @RequireRole(value = {77})  // 覆盖：仅系统管理员
    public ResponsePojo<List<User>> getUsers() {
        // 更严格的权限控制
    }
    
    @GetMapping("/statistics")
    @RequireRole(value = {55, 66, 77})  // 覆盖：审核员、工单管理员、系统管理员
    public ResponsePojo<Statistics> getStatistics() {
        // 放宽权限，多个角色可访问
    }
}
```

## 角色说明

### 角色代码定义

| 角色代码 | 角色名称   | 权限描述                | 典型应用场景           |
|:----:|--------|---------------------|------------------|
|  11  | 普通用户   | 浏览、评论、点赞、收藏        | 查看作品、发表评论、点赞收藏  |
|  22  | 创作者    | 发布作品、管理自己的作品       | 上传作品、编辑作品、删除作品  |
|  55  | 审核员    | 审核作品、处理举报          | 内容审核、违规处理       |
|  66  | 工单管理员  | 处理用户工单、客服支持       | 工单管理、用户问题处理     |
|  77  | 系统管理员  | 全部权限（用户管理、系统配置等） | 用户管理、系统配置、数据备份 |

### 角色层级关系

```
权限等级：11 < 22 < 55 < 66 < 77

当 allowHigher = true 时：
- @RequireRole({22}) → 22, 55, 66, 77 都可以访问
- @RequireRole({55}) → 55, 66, 77 都可以访问
- @RequireRole({77}) → 只有 77 可以访问

当 allowHigher = false 时：
- @RequireRole({22}, allowHigher=false) → 只有 22 可以访问
- @RequireRole({55}, allowHigher=false) → 只有 55 可以访问
```

### 角色分配

用户角色存储在 `tb_user` 表的 `user_role` 字段中，由系统管理员分配：

```sql
-- 查看用户角色
SELECT user_id, username, user_role FROM tb_user;

-- 修改用户角色（需管理员权限）
UPDATE tb_user SET user_role = 77 WHERE user_id = 1;
```

## 常见问题

### Q1: 如何判断应该使用类级别还是方法级别注解？

**A:** 
- **类级别**：当控制器中大部分接口需要相同权限时使用
- **方法级别**：当只有少数接口需要特殊权限控制时使用
- **混合使用**：类级别设置默认权限，方法级别覆盖特殊情况

### Q2: 权限验证失败会返回什么？

**A:** 返回 HTTP 403 状态码，响应格式：

```json
{
  "code": 403,
  "data": null,
  "message": "权限不足，需要角色:[77]"
}
```

### Q3: 没有 @RequireRole 注解的接口谁能访问？

**A:** 所有已通过 JWT 认证的用户都可以访问（即拥有有效 Token 的用户）。

### Q4: 如何实现数据级权限（如只能删除自己的作品）？

**A:** `@RequireRole` 只控制接口级权限，数据级权限需要在 Service 层实现：

```java
@Service
public class WorkServiceImpl implements WorkService {
    
    public Boolean deleteWork(Integer workId, Integer userId) {
        Work work = workMapper.selectById(workId);
        
        // 数据级权限验证
        if (!work.getUserId().equals(userId)) {
            log.warn("无权删除他人作品 - 作品ID:{}, 用户ID:{}", workId, userId);
            return false;
        }
        
        return workMapper.deleteById(workId) > 0;
    }
}
```

### Q5: 权限验证会影响性能吗？

**A:** 影响很小：
- 只对标注了 `@RequireRole` 的方法进行权限检查
- 用户角色信息可以通过 Redis 缓存优化
- 实测单次权限验证耗时 < 5ms

### Q6: 如何动态修改用户权限？

**A:** 修改数据库中 `tb_user` 表的 `user_role` 字段即可，下次请求时自动生效：

```sql
UPDATE tb_user SET user_role = 77 WHERE user_id = 1;
```

如果需要立即生效，可以清除该用户的 Token 白名单，强制重新登录。

### Q7: 如何添加新的角色类型？

**A:** 
1. 在数据库中添加新的角色代码
2. 在代码注释中更新角色说明
3. 在需要使用该角色的接口上添加 `@RequireRole` 注解

```java
// 新增角色：88 - 超级管理员
@RequireRole(value = {88})
public ResponsePojo<Void> superAdminAction() {
    // ...
}
```

### Q8: Token 过期后权限验证会怎样？

**A:** JWT 认证拦截器会先拦截，返回 401 错误，不会进入权限验证拦截器。

## 最佳实践

### 1. 最小权限原则

只授予用户完成工作所需的最小权限：

```java
// ❌ 不推荐：过度授权
@RequireRole(value = {77})  // 给所有管理员功能
public ResponsePojo<Void> simpleAction() { ... }

// ✅ 推荐：精确授权
@RequireRole(value = {22})  // 只给创作者
public ResponsePojo<Void> uploadWork() { ... }
```

### 2. 明确权限声明

始终显式声明权限需求，避免依赖默认行为：

```java
// ❌ 不推荐：依赖默认行为
@GetMapping("/sensitive-data")
public ResponsePojo<Data> getSensitiveData() { ... }

// ✅ 推荐：明确声明
@GetMapping("/sensitive-data")
@RequireRole(value = {77})
public ResponsePojo<Data> getSensitiveData() { ... }
```

### 3. 合理使用权限继承

根据业务逻辑决定是否允许更高权限访问：

```java
// 允许更高权限：管理员可以执行创作者的操作
@RequireRole(value = {22}, allowHigher = true)
public ResponsePojo<Void> uploadWork() { ... }

// 不允许更高权限：某些操作必须特定角色执行
@RequireRole(value = {55}, allowHigher = false)
public ResponsePojo<Void> reviewContent() { ... }
```

### 4. 结合业务层权限验证

接口级权限 + 数据级权限双重保障：

```java
// Controller 层：接口级权限
@PostMapping("/delete")
@RequireRole(value = {22, 77})
public ResponsePojo<Boolean> deleteWork(HttpServletRequest request, @RequestParam Integer id) {
    Integer userId = JWTUtils.getUserIdFromToken(request.getHeader("Authorization"));
    return workService.deleteWork(id, userId);
}

// Service 层：数据级权限
public Boolean deleteWork(Integer workId, Integer userId) {
    Work work = workMapper.selectById(workId);
    if (!work.getUserId().equals(userId)) {
        return false;  // 不能删除他人的作品
    }
    return workMapper.deleteById(workId) > 0;
}
```

### 5. 完善的日志记录

记录所有权限验证结果，便于审计和排查：

```java
// 权限拦截器已自动记录
log.warn("权限不足 - 用户ID:{}, 用户名:{}, 用户角色:{}, 需要角色:{}, URI:{}", 
    userId, username, userRole, Arrays.toString(allowedRoles), requestURI);
```

### 6. 统一的错误响应

权限验证失败返回统一的 403 错误格式：

```json
{
  "code": 403,
  "data": null,
  "message": "权限不足，需要角色:[77]"
}
```

### 7. API 文档同步

在 Swagger 文档中明确说明权限要求：

```java
@Operation(
    summary = "删除作品", 
    description = "创作者可删除自己的作品，审核员和管理员可删除任意作品\n\n**权限要求**：创作者(22)、审核员(55)、系统管理员(77)"
)
@RequireRole(value = {22, 55, 77})
public ResponsePojo<Boolean> deleteWork(...) { ... }
```

### 8. 定期审查权限配置

- 定期检查哪些接口使用了 `@RequireRole`
- 确认权限配置是否符合业务需求
- 清理不再使用的权限配置

## 附录

### A. 角色代码速查表

```
11 - 普通用户   (浏览、评论、点赞)
22 - 创作者     (发布、管理作品)
55 - 审核员     (审核内容、处理举报)
66 - 工单管理员  (处理用户工单)
77 - 系统管理员  (全部权限)
```

### B. 常用权限配置模板

```java
// 公开接口（无需登录）
// 不添加任何注解，并在 WebConfig 中排除

// 已登录用户可访问
// 不添加 @RequireRole 注解

// 创作者及以上
@RequireRole(value = {22})

// 审核员及以上
@RequireRole(value = {55})

// 工单管理员及以上
@RequireRole(value = {66})

// 仅系统管理员
@RequireRole(value = {77})

// 多角色组合
@RequireRole(value = {22, 55, 77})

// 不允许权限继承
@RequireRole(value = {55}, allowHigher = false)
```
