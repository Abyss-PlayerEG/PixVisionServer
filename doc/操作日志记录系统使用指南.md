# 操作日志记录系统使用指南

## 📖 概述

本项目提供了一套基于 AOP 切面的操作日志记录系统，可以自动将用户的操作行为记录到数据库 `tb_sys_logs` 表中。通过简单的注解标记，即可实现无侵入式的日志记录。

---

## 🏗️ 架构设计

### 核心组件

```
@LogRecord 注解
    ↓
LogAspect 切面（拦截标注的方法）
    ↓
提取用户 ID（从 JWT Token）
    ↓
OperateLogService 服务层
    ↓
OperateLogMapper 数据访问层
    ↓
tb_sys_logs 数据库表
```

### 文件结构

```
src/main/java/top/playereg/pix_vision/
├── util/Aspect/
│   ├── LogRecord.java          # 日志记录注解
│   └── LogAspect.java           # AOP 切面实现
├── service/
│   ├── OperateLogService.java           # 日志服务接口
│   └── Impl/OperateLogServiceImpl.java  # 日志服务实现
└── mapper/
    ├── OperateLogMapper.java            # Mapper 接口
    └── resources/mapper/OperateLogMapper.xml  # SQL 映射
```

## 🚀 快速开始

### 1. 基本用法

在需要记录日志的 Controller 方法上添加 `@LogRecord` 注解：

```java
import top.playereg.pix_vision.util.Annotation.LogRecord;

@RestController
@RequestMapping("/api/user")
public class UserProfileController {

    @PostMapping("/nickname")
    @LogRecord(module = "用户管理", event = "修改昵称")
    public ResponsePojo<Boolean> updateNickname(
      HttpServletRequest request,
      @RequestParam String nickname
    ) {
        // 业务逻辑...
        return ResponsePojo.success(true, "昵称修改成功");
    }
}
```

### 2. 注解参数说明

| 参数 | 类型 | 必填 | 说明 | 示例 |
|------|------|------|------|------|
| `module` | String | 否 | 模块名称 | `"用户管理"`、`"作品管理"` |
| `event` | String | 否 | 事件描述 | `"修改昵称"`、`"上传作品"` |

**最终记录格式：** `[模块] 事件`  
例如：`[用户管理] 修改昵称`

### 3. 完整示例

```java
/**
 * 删除用户拓展数据
 */
@DeleteMapping("/data")
@LogRecord(module = "用户数据", event = "批量删除拓展数据")
public ResponsePojo<Boolean> deleteUserData(
    HttpServletRequest request,
    @RequestBody List<Integer> dataIds
) {
    // 提取 Token
    String token = JWTUtils.extractToken(request);
    
    // 检查白名单
    if (!tokenWhitelistService.isInWhitelist(token)) {
        return ResponsePojo.error(null, "Token 已失效");
    }
    
    // 获取用户 ID
    Integer userId = JWTUtils.getUserIdFromToken(token);
    
    // 执行删除操作
    Boolean result = userService.batchDeleteUserData(userId, dataIds);
    
    // 返回结果（日志会自动记录）
    return result 
        ? ResponsePojo.success(true, "删除成功")
        : ResponsePojo.error(false, "删除失败");
}
```

## 💾 数据库表结构

### tb_sys_logs 表

```sql
CREATE TABLE `tb_sys_logs` (
  `sys_log_id` int NOT NULL AUTO_INCREMENT COMMENT '唯一值',
  `user_id` int NOT NULL COMMENT '操作者id',
  `log_datetime` timestamp NOT NULL COMMENT '操作日期时间',
  `log_event` varchar(64) CHARACTER SET utf8mb4 COLLATE utf8mb4_german2_ci NOT NULL COMMENT '操作事件',
  PRIMARY KEY (`sys_log_id`),
  KEY `tb_sys_logs_tb_user_FK` (`user_id`),
  CONSTRAINT `tb_sys_logs_tb_user_FK` FOREIGN KEY (`user_id`) REFERENCES `tb_user` (`user_id`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_german2_ci COMMENT='系统操作日志';
```

### 字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| `sys_log_id` | int | 日志 ID（自增主键） |
| `user_id` | int | 操作用户 ID（外键关联 tb_user） |
| `log_datetime` | timestamp | 操作时间（自动记录） |
| `log_event` | varchar(64) | 操作事件描述 |

## 🔧 工作原理

### 1. 用户 ID 提取流程

```
请求到达 → LogAspect 拦截
    ↓
优先从 request.getAttribute("userId") 获取
    ↓
如果没有，从 JWT Token 中解析
    ↓
提取成功后记录日志
```

**代码实现：**
```java
private Integer extractUserIdFromRequest(HttpServletRequest request) {
    // 优先从 request attribute 中获取（由拦截器设置）
    Integer userId = (Integer) request.getAttribute("userId");
    if (userId != null && userId > 0) {
        return userId;
    }

    // 如果 attribute 中没有，尝试从 Token 中解析
    String token = JWTUtils.extractToken(request);
    if (token != null && !token.isEmpty()) {
        return JWTUtils.getUserIdFromToken(token);
    }

    return null;
}
```

### 2. 日志记录时机

- **环绕通知（@Around）**：在目标方法执行**之后**记录日志
- **异常安全**：即使日志记录失败，也不会影响业务逻辑

### 3. 事件描述构建

```java
private String buildLogEvent(String module, String event) {
    StringBuilder sb = new StringBuilder();
    if (module != null && !module.isEmpty()) {
        sb.append("[").append(module).append("] ");
    }
    if (event != null && !event.isEmpty()) {
        sb.append(event);
    } else {
        sb.append("未知操作");
    }
    return sb.toString();
}
```

**示例输出：**
- `[用户管理] 修改昵称`
- `[作品管理] 上传作品`
- `未知操作`（当两个参数都为空时）

## ⚠️ 注意事项

### 1. Token 要求

- 标注了 `@LogRecord` 的接口**必须携带有效的 Token**
- 如果无法提取用户 ID，日志将**不会记录**（但业务逻辑正常执行）

### 2. 性能考虑

- 日志记录在方法执行**之后**进行，不影响响应速度
- 数据库插入操作是异步安全的，失败不会影响业务
- 建议不要在高频率调用的接口上使用（如列表查询）

### 3. 适用场景

✅ **推荐使用：**
- 数据修改操作（增删改）
- 敏感操作（密码修改、权限变更）
- 关键业务流程节点

❌ **不推荐使用：**
- 高频查询接口（如列表分页）
- 公开接口（无 Token）
- 静态资源访问

### 4. 异常处理

```java
try {
    // 记录日志到数据库
    boolean success = operateLogService.recordLog(userId, logEvent);
    if (!success) {
        logger.warn("操作日志记录失败 - 用户 ID: {}, 事件: {}", userId, logEvent);
    }
} catch (Exception e) {
    // 日志记录失败不应影响业务逻辑，仅记录错误
    logger.error("日志记录过程中发生异常: {}", e.getMessage(), e);
}
```

**保证：** 即使日志记录失败，原始方法的返回值和异常行为不受影响。

## 🔍 常见问题

### Q1: 为什么日志没有记录到数据库？

**可能原因：**
1. 没有携带有效的 Token
2. Token 已过期或不在白名单中
3. 用户 ID 提取失败
4. 数据库连接异常

**排查步骤：**
```java
// 1. 检查应用日志
grep "操作日志" logs/pix_vision.log

// 2. 确认 Token 有效
curl -H "Authorization: Bearer YOUR_TOKEN" http://localhost:9090/7e212056/require-auth

// 3. 检查数据库连接
SHOW TABLES LIKE 'tb_sys_logs';
```

### Q2: 如何自定义日志记录逻辑？

可以直接调用 Service 层：

```java
@Autowired
private OperateLogService operateLogService;

public void customOperation() {
    // 业务逻辑...
    
    // 手动记录日志
    operateLogService.recordLog(userId, "[自定义模块] 自定义事件");
}
```

### Q3: 能否记录方法参数和返回值？

当前版本仅记录基础信息（用户 ID、时间、事件）。如需记录详细信息，可以：

1. 扩展 `OperateLog` 实体类，添加更多字段
2. 在 `LogAspect` 中通过 `pjp.getArgs()` 获取参数
3. 在 `LogAspect` 中通过 `result` 获取返回值

**示例：**
```java
@Around("@annotation(logRecord)")
public Object record(ProceedingJoinPoint pjp, LogRecord logRecord) throws Throwable {
    // 获取方法参数
    Object[] args = pjp.getArgs();
    logger.debug("方法参数: {}", Arrays.toString(args));
    
    // 执行方法
    Object result = pjp.proceed();
    
    // 获取返回值
    logger.debug("方法返回值: {}", result);
    
    // 记录日志...
    return result;
}
```

### Q4: 日志记录会影响性能吗？

**影响极小：**
- 仅在方法执行后进行一次数据库 INSERT 操作
- 使用 try-catch 包裹，异常不影响业务
- 建议配合数据库索引优化查询性能

**优化建议：**
```sql
-- 为常用查询字段添加索引
CREATE INDEX idx_user_id ON tb_sys_logs(user_id);
CREATE INDEX idx_log_datetime ON tb_sys_logs(log_datetime);
```
